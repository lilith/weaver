# Weaver — Inline Script (DEPRECATED)

> **Status: DEPRECATED as of 2026-04-19.** The three-execution-paths split was collapsed to two: JSON-with-safe-inline-expressions + modules. See `01_ARCHITECTURE.md` §"Two execution paths, one store" and `00_OVERVIEW.md` principle #2.
>
> **Do not build a separate inline-script interpreter.** Do not ship the grammar below. The use cases described here are now served either by (a) inline expressions inside the template grammar of `02_LOCATION_SCHEMA.md` (for conditional prose, RNG flavor, small per-player state writes) or (b) a durable module (for anything truly stateful, multi-step, or cross-turn).
>
> This file is kept for historical reference and as the source of the use-cases list that informed the collapse. It is not authoritative for implementation.

## (Historical) Purpose

For the 4% of locations where JSON templating isn't expressive enough but a full durable module is overkill. Examples:
- Small random encounters: "30% chance a fox appears." — **today:** an inline expression in the template or an `on_enter` effect with `{kind: "roll", sides: 100, save_as: "roll"}` + a conditional option.
- Conditional multi-step dialogue within one location. — **today:** a module with 2-3 steps.
- Dice-driven outcomes with stateful tally. — **today:** a module; tally lives in the flow's state blob.

## Design constraints

- Pure function: `(state, input?) → (render_ops, new_state)`.
- No `eval`, no `Function`, no dynamic property access on native prototypes.
- No I/O, no network, no imports.
- Deterministic given `rng(seed)` provided by runtime.
- ~300 LOC interpreter.
- Authored by humans in browser designer or generated by AI.
- Validates statically: all variable reads must resolve to declared scope paths.

## Surface

Looks like Weaver's original Lua with minor modernization.

### Top-level ops

```
p "some text"                    # print (inline description)
p("formatted {{var}}")           # print with interpolation
p[[
  multi-line text
]]                               # multi-line

message "text"                   # show a modal "continue" message, blocks

choose {                         # present options, blocks until user picks
  "label 1": target_or_block,
  "label 2": target_or_block,
}

goto "location_id"               # move player there
goto_inline "other_script_name"  # jump to another inline script on same entity

set path = value                 # write to scope
inc path by n                    # increment
roll sides into var              # RNG, save into local var

if cond then ... elseif cond then ... else ... end
while cond do ... end            # bounded: max 1000 iterations, else runtime error
```

### Expressions

- Literals: numbers, strings, booleans, `nil`.
- Paths: `character.x`, `this.y`, `location.z`, `world.w`.
- Comparisons: `==`, `!=`, `<`, `<=`, `>`, `>=`.
- Boolean ops: `and`, `or`, `not`.
- Method calls (whitelist only): `.has()`, `.length`, `.count()`, `.contains()`.
- String concat: `"foo " + var`.
- Arithmetic: `+`, `-`, `*`, `/`, `%`.

### Locals

```
local x = 5
local msg = "hello " + character.name
```

Locals scope to the script invocation. Disappear when script ends. To persist, assign to `this.`, `location.`, `character.`, or `world.` scope.

### Effects as first-class

Effects match location schema effects:

```
give_item "snowglobe"
take_item "snowglobe"
start_combat "road_bandit"
add_predicate "fed_doe" to npc.doe_42 with {bond: 0.3}
emit "needs_expansion" with {hint: "..."}
```

## Worked examples

### Example 1 — Merchant snowglobe arc

```
# script: merchant_booth_script

p "You approach the odd merchant and examine each trinket individually."
p "A finely-crafted snowglobe catches your eye."

if this.examined_globe then
  p "The merchant recognizes you. \"Back for another look?\""
else
  set this.examined_globe = true
end

choose {
  "Pick up the snowglobe": pickup_globe,
  "Leave": leave,
}

# -----

function pickup_globe()
  p[[
    Curious, you lift the globe. The merchant's voice, low and muffled:
    "That there is a magical artifact. Shake it and something...
    special may happen."
  ]]

  if character.gold < 500 then
    p "He sees your face fall at the price. \"500 coins. Come back when you have them.\""
    goto_inline "leave"
  end

  choose {
    "Pay the merchant 500 coins": pay,
    "Put it down and leave": leave,
  }
end

function pay()
  inc character.gold by -500
  give_item "snowglobe"
  p[[
    The merchant's fingers count the coins. The snowglobe is yours.
    "Shake it when you're ready," he says.
  ]]
  done
end

function leave()
  message "You politely nod your head as you leave the man's booth."
  goto "market_center"
end
```

### Example 2 — Random forest encounter

```
# script: forest_traversal_encounter

roll 100 into chance

if chance <= 30 then
  p "A red fox pauses on the path, regarding you with bright eyes."
  choose {
    "Approach slowly": approach_fox,
    "Continue past": continue,
  }
elseif chance <= 40 then
  p "A small bird alights on a branch and sings three clear notes, then is gone."
  continue
else
  p "The path is quiet. The forest watches."
  continue
end

function approach_fox()
  roll 20 into reaction
  if reaction >= 15 then
    p "The fox allows you close. For a moment, you touch its fur, then it is gone."
    add_predicate "met_fox" to world with {on_day: world.day}
  elseif reaction >= 5 then
    p "The fox holds your gaze, then slips into the undergrowth."
  else
    p "The fox bolts before you take two steps."
  end
  continue
end

function continue()
  goto "forest_next"
end
```

### Example 3 — Tally-based dialogue

```
# script: elder_riddles
# The elder asks three riddles; answers tallied; outcome varies.

if this.tally == nil then
  set this.tally = 0
  set this.question = 1
end

if this.question == 1 then
  p "\"What has roots as nobody sees?\" the elder asks."
  choose {
    "\"A mountain\"": right,
    "\"A tree\"": wrong,
    "\"I don't know\"": wrong,
  }
elseif this.question == 2 then
  p "\"Voiceless it cries, wingless flutters...\""
  choose {
    "\"The wind\"": right,
    "\"A ghost\"": wrong,
    "\"I don't know\"": wrong,
  }
elseif this.question == 3 then
  p "\"This thing all things devours...\""
  choose {
    "\"Time\"": right,
    "\"Fire\"": wrong,
    "\"I don't know\"": wrong,
  }
else
  goto_inline "finalize"
end

function right()
  inc this.tally by 1
  inc this.question by 1
  goto_inline "elder_riddles"
end

function wrong()
  inc this.question by 1
  goto_inline "elder_riddles"
end

function finalize()
  if this.tally >= 2 then
    p "The elder smiles. \"You may pass.\""
    give_item "elder_blessing"
    goto "sacred_grove"
  else
    p "The elder shakes his head. \"Return when you've listened more.\""
    goto "village_square"
  end
end
```

## Parser / evaluator

Implementation plan:

1. **Lexer** (~80 LOC): tokenize source into keywords, identifiers, literals, operators. Handle `[[...]]` multi-line strings.
2. **Parser** (~120 LOC): recursive descent to AST. Top level is sequence of ops; function declarations collected separately.
3. **Static validator** (~50 LOC): traverse AST, collect all path reads/writes, verify against declared `state_keys` + whitelisted methods. Ensure all `goto_inline` targets exist. Reject on any violation.
4. **Evaluator** (~100 LOC): interpret AST against injected `InlineCtx` (similar to `ModuleCtx` but restricted). Step-limited (10K steps/invocation) to prevent runaway.

```ts
interface InlineCtx {
  get: (path: string) => unknown
  set: (path: string, value: unknown) => void
  inc: (path: string, by: number) => void
  rng: (label: string) => number
  say: (text: string) => void
  choose: (opts: Record<string, string>) => Promise<string>
  goto: (target: string) => void
  effect: (e: LocationEffect) => void
  done: () => void
}
```

Evaluation is async; `choose` returns the user's pick, `message` blocks on user "continue." Behind the scenes the evaluator suspends and resumes via the durable runtime (same mechanism as modules), but authors never see that complexity.

## Validation gates

Per inline script at insert/update:

1. Parse successfully.
2. Static validator passes (all paths declared, all targets resolvable, no suspicious constructs).
3. Bounded execution check: a linter flags unbounded `while` loops without a provable exit.
4. Property-based test pass: runtime auto-generates 100 random valid inputs, asserts no crashes, no negative resources, no orphan state references.

## AI generation of inline scripts

When Opus generates an inline script from a prompt, it receives:
- World bible (cached).
- Parent location JSON for context.
- The grammar spec (this document, trimmed).
- A small set of few-shot examples.

Output: raw script text. Validated through the gates above before insert. On failure, up to 2 retry rounds with error feedback in the prompt.

## Author UX in browser

The module designer's form has a "custom script" option that drops into a code editor (CodeMirror 6, syntax-highlighted for Weaver grammar). "Test in sandbox" button runs the script against mock state and shows the rendered output. "Suggest with prompt" lets the author describe what they want in English; Opus generates, author reviews diff, accepts or edits.

All authored scripts carry `author_pseudonym` and get versioned like every other artifact. Edit-with-prompt works on scripts the same way it works on JSON — described in `11_PROMPT_EDITING.md`.
