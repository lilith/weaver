# Weaver — Authoring Source Format and Sync

*For Claude Code (and any other filesystem-based authoring agent) participating in creative work on a Weaver world.*

## Scope and philosophy

The **DB is the runtime source of truth** for every Weaver world. All live state — entities, components, relations, events, chat, flows, art, themes — lives in Convex. Prompt-edits and expansion-loop creations write directly to the DB. Nothing in this document changes that.

This document describes a **filesystem mirror** that exists purely so Claude Code and other coding agents can do creative work against a Weaver world:

- Bootstrap a new world from a directory of authored files.
- Propose new locations, characters, biomes, items against an existing world.
- Review and refactor existing content (fix tone drift, consolidate near-duplicate locations, tighten a biome's voice).

Flow is **explicit, human-gated, not real-time**:

- **Export** (DB → files): on demand via CLI, plus a nightly cron backup. Produces a snapshot directory.
- **Validate**: a Zod-backed checker runs in three contexts (author-time, pre-import, pre-export). Same schemas everywhere.
- **Import** (files → DB): on demand via CLI with an explicit collision mode (`--mode=new|edit`). Never automatic.

Git is not in the pipeline. If you want to version-control a snapshot directory, `git init` it yourself. The system doesn't care.

## File layout

One source directory per world. Layout:

```
my-world/
├── .weaver-sync/
│   ├── world.json          # world + branch ids, schema versions, timestamps
│   └── map.json            # slug → entity_id bindings (generated by importer)
├── bible.md                # world bible (one per world)
├── biomes/
│   └── <slug>.md
├── characters/             # player characters + travelling NPCs
│   └── <slug>.md
├── locations/
│   └── <slug>.md
├── npcs/                   # location-bound NPCs
│   └── <slug>.md
├── items/
│   └── <slug>.md
├── refs/                   # canonical reference images (style, biome, character)
│   └── <slug>.md
├── modules/                # durable modules (Path 2, TypeScript)
│   └── <slug>/
│       ├── manifest.json
│       └── steps.ts
└── themes/
    └── current.json
```

- **One entity per file.** Filename stem becomes the stable slug.
- **Slugs** are lowercase-hyphenated ASCII. `forest-clearing-42`, `violet-the-barmaid`, `snowglobe`.
- **`.weaver-sync/`** is machine-written. Don't hand-edit; treat as opaque.

## File format

Two conventions, chosen per entity type by what reads best:

### Convention A — YAML frontmatter + markdown body

Used for any entity whose primary payload is prose. The body fills a specific named field (usually `description_template` or `description`).

```markdown
---
<structured fields in YAML>
---

<prose body that becomes description_template or description>
```

Used for: **bible, biome, character, location, npc, item**.

### Convention B — pure JSON

Used for entities with no prose component.

Used for: **themes, refs** (when the payload is a blob reference + metadata, not prose).

### Convention C — module directories

Durable modules (Path 2) live in `modules/<slug>/`:

```
modules/merchant-arc/
├── manifest.json          # declared reads/writes/emits, schema_version, entrypoint step
└── steps.ts               # TypeScript ModuleDef — step handlers as typed functions
```

The module is imported by the runtime as compiled TypeScript (Wave 1-3 trusted-code model; see `01_ARCHITECTURE.md` §"Module context"). The importer validates the manifest shape, type-checks `steps.ts` against the `ModuleDef` type, and registers the module handlers in the runtime.

Inline-script files (`scripts/<slug>.weaver`) are deprecated — see `03_INLINE_SCRIPT.md`. Content from prior drafts that used inline scripts should be ported to either inline expressions in the JSON template or a small module.

## What's in files vs. DB-only

Every entity has authored fields (round-trip through files) and runtime fields (DB-only). The importer ignores runtime fields in files; the exporter omits them.

**Round-trips to files:**
- Authored payload (prose, options, effects, tone, voice samples, etc.)
- Slug / stable identity
- `author_pseudonym` (not `author_user_id`)
- `tags`
- Relationships to other authored entities (by slug)
- Blob-hash references for refs / art / themes (content-addressed, stable)

**DB-only, never in files:**
- Convex `_id` values
- `created_at`, `updated_at`, `version`, `schema_version`
- `author_user_id` (files carry pseudonym only)
- `art_status`, queued-generation state
- `chat_thread_id` (created lazily in DB)
- `discovered_by` user_id
- Event log, flow stack, any runtime-materialized state
- Chat messages, presence, cost ledger, mentorship log

The full mapping is the validator's responsibility — see §Validator contract.

## Per-type reference

Schemas are authoritative in `02_LOCATION_SCHEMA.md` (locations) and `05_WORLD_BIBLE_BUILDER.md` (bible). This section defines the **file layer** on top of those: what goes in frontmatter, what goes in the body, which fields are authored vs. runtime.

### bible.md

Convention A. One per world.

```markdown
---
name: The Quiet Vale
tagline: A small mountain valley in early spring, watched over by old gods.
content_rating: family
creativity: balanced
tone:
  descriptors: [cozy, gentle, slightly_whimsical, reverent]
  avoid: [grimdark, cynical, body_horror, preachy]
style_anchor:
  descriptor: cozy watercolor, soft ink lines, warm muted palette
  ref: style-anchor-cottage-dusk     # slug of refs/style-anchor-cottage-dusk.json
  prompt_fragment: "cozy watercolor, soft ink outlines, warm palette"
biomes: [forest, village, inn, chapel-ruins]
characters: [mara, jason, old-halvard]
established_facts:
  - Magic returned to the world five years ago.
  - The old king died childless; the Vale has no central ruler.
  - Dogs can speak here, but only when no one's listening.
taboos:
  - Violence against children.
  - Real-world brands or celebrities.
  - Anything that would frighten Jason (age 8).
---

# Voice sample

The wind moved through the birches with an old kind of patience.
Mara pulled her cloak tighter and watched her breath rise. Somewhere
downhill, a kettle was starting to sing.

# Themes and motifs

- Small kindnesses matter more than grand gestures.
- The world is recovering; ruin is visible but not central.
- Animals and weather are half-characters.
```

The body's `# Voice sample` section becomes `tone.prose_sample`. Other `#` sections are stored as free-form annotations passed to prompt generation.

### biomes/\<slug\>.md

Convention A.

```markdown
---
name: The birch forest
tags: [outdoor, wild, diurnal]
establishing_shot_prompt: |
  A wide view of a birch forest in late afternoon, gentle low sun through
  the trunks, a faint path winding out of frame, no characters.
ref: biome-ref-forest               # slug of refs/biome-ref-forest.json, set after art gen
---

Cool, quiet, pale-trunked birches standing in loose ranks. Mossy ground,
patches of sunlight. Streams are frequent and small. Nothing dangerous,
but the woods hide things — old stone cairns, older things beneath them.
```

Body becomes `description`. `establishing_shot_prompt` is what the art queue uses to generate the ref image.

### characters/\<slug\>.md

Convention A.

```markdown
---
name: Mara
pseudonym: Mara
role: player_character                # player_character | core_npc | pet
tags: [human, adult, woodworker]
always_wears: [a green cloak, a silver ring on the right hand]
linked_user: null                     # set by importer if pseudonym matches a user
portrait_prompt: |
  A 3/4 portrait of a young woman, late twenties, short dark hair, green
  cloak with a silver ring visible on her right hand, quiet watchful
  expression, cozy watercolor style.
refs: []                              # populated after portrait+sheet gen
voice:
  style: Terse. Doesn't soften statements. Dry humor when she does joke.
  examples:
    - "The roof leaks. We fix it or we move."
    - "I'll be back by dusk."
relationships:
  - with: old-halvard
    note: Respects him but thinks he's wrong about the old gods.
  - with: jason
    note: Treats him like a younger brother; patient, slightly amused.
---

Mara is in her late twenties, small, watchful. Grew up in the Vale, left
for a decade, came back last spring. Carpenter by trade. Speaks only when
she has something to say. Walks with a very slight limp nobody remembers
the cause of.
```

Body becomes `description`. `refs` starts empty; the importer enqueues portrait generation and fills in ref slugs after.

### locations/\<slug\>.md

Convention A. The highest-volume type; see `02_LOCATION_SCHEMA.md` for the canonical schema.

```markdown
---
name: A small clearing
biome: forest
coords: { q: 4, r: -2 }
neighbors:
  n: forest-deep
  s: village-square
  e: forest-creek
tags: [has_chat]
safe_anchor: false
author_pseudonym: Stardust
state_keys: [this.visited, this.stump_examined, world.weather.rain]
on_enter:
  - { kind: inc, path: this.visited, by: 1 }
options:
  - label: Examine the old stump
    effect:
      - { kind: set, path: this.stump_examined, value: true }
      - { kind: say, text: "Rings uncountable. Older than any living memory." }
  - label: Head deeper into the forest
    target: forest-deep
  - label: Return to the village
    target: village-square
  - label: Rest by the stump
    condition: "character.energy < 5"
    effect:
      - { kind: inc, path: character.energy, by: 3 }
---

You stand in a small clearing ringed by pale birches.
{{#if world.weather.rain}}Rain taps softly against the leaves.{{/if}}
{{#if this.visited}} The path back to the village curves behind you.{{/if}}
An old stump occupies the center.
```

Body becomes `description_template`. The template grammar is the real engine grammar (`{{#if ...}}`, `{{var.path}}`, etc.) from `02_LOCATION_SCHEMA.md` — no shorthand.

`target` values reference other authored entities by slug. The importer resolves slugs to Convex IDs. An unresolved slug during `--mode=new` import creates a stub location, same as the expansion loop does (flagged in the validator output so you see it).

`#module:<name>/<step>` targets work in files — the importer resolves module names by slug against `modules/`. `#inline:` targets are deprecated (see `03_INLINE_SCRIPT.md`).

Convention A. Simpler than characters — NPCs don't travel.

```markdown
name: Violet
lives_at: inn-common-room

  - "Back again! You'll wear a path in my floorboards."
  - "Have you seen Halvard today? He's late."

A middle-aged woman with flour-dusted forearms and a ready laugh. She's
been pouring drinks at this inn for twenty years and has opinions about
everyone who's ever passed through.
```

### items/\<slug\>.md

Convention A.

```markdown
---
name: Snowglobe of the Ice Realm
tags: [artifact, fragile, magic]
stackable: false
portrait_prompt: |
  A small snowglobe on a pewter base, a tiny detailed pine tree inside

---

A heavy glass sphere on a pewter base. Inside, a tiny snow-covered
pine stands impossibly sharp and detailed. If you shake it, the snow

```

### refs/\<slug\>.json

Convention B. A canonical reference image (style anchor, biome anchor, character portrait, item illustration). The importer may generate these from `*_prompt` fields on other entities; writing a ref file directly is for cases where you already have a hand-curated image.

```json
{
  "slug": "style-anchor-cottage-dusk",
  "kind": "style_anchor",
  "prompt": "A small cottage in a clearing at dusk, warm window light, cozy watercolor",
  "blob_hash": "sha256-a1b2c3d4...",
  "source_url": "https://art.theweaver.quest/a1b2c3d4.webp",
  "width": 1024,
  "height": 1024,
  "created_by": "Stardust"
}
```

`blob_hash` is the content-addressed key into the blob store (`12_BLOB_STORAGE.md`). If you don't have one yet, leave it `null` — the importer enqueues generation from `prompt`.

### modules/\<slug\>/

Convention C. A module directory with a manifest and a TypeScript steps file.

```json
// modules/merchant-arc/manifest.json
{
  "slug": "merchant-arc",
  "schema_version": 1,
  "entrypoint": "open",
  "reads":  ["character.inventory", "character.gold"],
  "writes": ["character.inventory", "character.gold"],
  "emits":  ["item_acquired"],
  "author_pseudonym": "Stardust",
  "tags": ["dialogue", "transaction"]
}
```

```ts
// modules/merchant-arc/steps.ts
import type { ModuleDef } from "@weaver/engine"

const mod: ModuleDef = {
  name: "merchant-arc",
  schema_version: 1,
  manifest: { reads: [...], writes: [...], emits: [...] },
  steps: {
    open: async (ctx, state) => { /* ... */ },
    /* ... */
    done: { terminal: true },
  },
}
export default mod
```

### themes/current.json

Convention B. Full theme JSON per `10_THEME_GENERATION.md`. Multiple themes per world are named `themes/<slug>.json`; the active one is pointed at by `bible.md`'s `theme:` field or defaulted to `current`.

## Validator contract

A single validator, `packages/engine/validate/`, consumed by three callers.

### Input

A source directory path (for CLI/pre-import) or a single in-memory entity object (for author-time checks).

### Output

```ts
type ValidationResult = {
  ok: boolean
  issues: Issue[]
  entities: Record<string, EntityValidated>  // slug → parsed entity, if ok
}

type Issue = {
  severity: "error" | "warning"
  kind:
    | "schema"                // Zod parse failure
    | "unresolved_slug"       // reference to missing entity
    | "template"              // description_template doesn't parse, or references undeclared state_key
    | "circular_ref"          // relationship graph has a cycle where none is allowed
    | "taboo_match"           // heuristic match against bible's taboos list
    | "rating_violation"      // content appears to exceed declared rating
    | "orphan"                // entity unreferenced from any other (warning only)
    | "duplicate_slug"        // same slug in two files
    | "frontmatter_shape"     // unknown or missing required fields
  file: string                // relative path
  line?: number
  message: string
  suggestion?: string         // machine-actionable fix if available
}
```

### Checks performed

1. **Zod schema parse** for each entity against its type's schema. The same Zod definitions live in `packages/engine/schemas/`; they're shared between runtime and the validator.
2. **Template compile.** Every `description_template` is parsed to AST. All `{{var}}` references must resolve against declared `state_keys` (for locations) or a known global (for bible, biome prose).
3. **Slug reference resolution.** Every `target`, `neighbor`, `ref`, `lives_at`, `linked_location`, `refs[]`, etc., points to a file that exists in the source directory OR to an existing entity in the DB (when validating against a loaded DB snapshot).
4. **Unique slugs.** No two files produce the same slug for the same entity type.
5. **Content rating check.** Haiku-based heuristic scan against `bible.taboos` and `content_rating`. Warning-level by default; error-level if `--strict`.
6. **Safety-anchor invariant.** At least one location has `safe_anchor: true`. Error at world level if not.
7. **Bible consistency.** Characters/biomes named in `bible.md` have files; no dangling references.

### Callers

- **CLI `weaver validate <dir>`** — prints human-readable report; exit code 0 if ok, 1 if errors, 2 if only warnings.
- **`weaver import`** pre-flight — refuses to import if any errors.
- **`weaver export`** pre-flight — validates before writing files, so a broken DB state produces a broken export that at least includes its own diagnostic.
- **Author-time API** — importable function `validateEntity(obj, kind)` that Claude Code can call between edits to self-check.

## Import

```
weaver import <dir> --mode=new|edit [--world <world-slug>] [--branch <branch-slug>] [--dry-run]
```

- **`--mode=new`** — treats every entity as newly authored. Error if any slug already exists in the target world. Used for bootstrapping a world from scratch, or for adding disjoint new content whose slugs haven't been taken.
- **`--mode=edit`** — every authored slug must already exist in the target world. Matches are treated as edits → bumps version, creates `artifact_versions` entry, updates payload blob. Unknown slugs are errors unless `--allow-new` is also passed.
- **`--dry-run`** — runs validator + diff against current DB, prints what *would* happen, writes nothing.
- Required flags: `--world` (and `--branch` if multi-branch). Defaults to the single-world, single-branch case if only one exists.

Import steps:

1. Read and validate the source directory. Errors abort.
2. Load or infer `world_id` / `branch_id` from `--world` / `--branch` or the single default.
3. For each entity, resolve slug → existing entity_id (from `.weaver-sync/map.json` or DB lookup). Match against `--mode`.
4. Produce a transaction plan: inserts, version-bumps, blob-hash updates, art-queue enqueues.
5. Apply in a single Convex transaction per world (or per-branch if cross-branch).
6. Write back updated `.weaver-sync/map.json` with any newly-assigned entity IDs.
7. Print a summary: N inserted, N edited, M art-gens queued, warnings list.

## Export

```
weaver export --world <world-slug> --out <dir> [--include-generated] [--branch <branch-slug>]
```

- Walks the DB for the world, emits files in the layout above.
- Pulls blob content via blob hashes where applicable; writes stable `blob_hash` fields, not signed URLs.
- **`--include-generated`** includes locations / items / scene art created during play (default: includes everything; pass `--authored-only` to exclude expansion-loop content).
- Writes a fresh `.weaver-sync/map.json` and `world.json` alongside content files.

### Nightly cron

Scheduled Convex action, runs daily at 03:00 local time:

```ts
// convex/scheduled/nightlyBackup.ts
export const nightlyBackup = action(async (ctx) => {
  const worlds = await ctx.db.query("worlds").collect()
  for (const w of worlds) {
    const outDir = `${process.env.WEAVER_BACKUP_DIR}/${w.slug}-${isoDate()}`
    await exportWorld(ctx, { world_id: w._id, out: outDir })
  }
})
```

`WEAVER_BACKUP_DIR` env var points to a local directory Lilith chooses. No remote upload by default — that's a user choice.

Retention policy: the exporter writes to `<dir>/<world>-YYYY-MM-DD/`, keeping one snapshot per day. A separate trivial script can prune old ones. The system does not auto-delete backups.

## Typical Claude Code workflows

### 1. Bootstrap a new world

Lilith asks Claude Code to scaffold a world from a concept.

```
$ mkdir -p worlds/quiet-vale
$ cd worlds/quiet-vale
# Claude Code authors bible.md, biomes/, characters/, locations/, etc.
$ weaver validate .
# Fix any errors Claude self-reports
$ weaver import . --mode=new --world quiet-vale
# World now exists in DB; art queue kicks off for refs and scene images
```

### 2. Extend an existing world

Lilith wants 12 new locations for an existing world. Fresh snapshot + authoring + import.

```
$ weaver export --world quiet-vale --out worlds/quiet-vale-snapshot
# Claude Code reads the snapshot, adds 12 new locations/*.md files
$ weaver validate worlds/quiet-vale-snapshot
$ weaver import worlds/quiet-vale-snapshot --mode=edit --allow-new --world quiet-vale
# New locations inserted as new; existing ones re-validated (unchanged ones no-op).
```

The `edit` mode with `--allow-new` is the "bring in new content without stomping existing" pattern. Claude Code doesn't touch existing files unless Lilith asks it to.

### 3. Refactor (tone-tighten an existing biome)

Lilith wants all forest locations to lean more "reverent, fewer jokes."

```
$ weaver export --world quiet-vale --out worlds/quiet-vale-snapshot
# Claude Code revises locations/*.md with biome=forest, tightening tone
$ weaver validate worlds/quiet-vale-snapshot
$ weaver import worlds/quiet-vale-snapshot --mode=edit --world quiet-vale
# Each edited location becomes a new version in artifact_versions.
# Old versions remain rollback-able.
```

### 4. Review without changing

Lilith wants Claude Code to review worldstate and point out inconsistencies. Export-only — no import.

```
$ weaver export --world quiet-vale --out worlds/quiet-vale-snapshot
# Claude Code reads, reports findings. No writeback.
```

## Collision and conflict handling

Between the nightly backup and the next manual export, the DB may have drifted (prompt-edits, new locations from expansion loop). When Lilith re-exports, Claude's in-flight file edits are now against a stale snapshot.

There's no real-time reconciliation — the model is human-gated. Pattern:

1. Claude's work in progress lives in a dedicated directory (`worlds/quiet-vale-wip/`).
2. Before importing, export a *fresh* snapshot to a sibling directory (`worlds/quiet-vale-snapshot/`).
3. Claude (or Lilith, or a diff tool) reconciles: compares Claude's WIP against the fresh snapshot, identifies which files Claude touched that have *also* changed in DB since the WIP started.
4. Those are the conflicts. Claude resolves per file: keep mine, keep DB's, manually merge.
5. Only after reconciliation does `weaver import` run.

This is slow compared to real-time sync — and that's the point. Authoring sessions are deliberate acts, not continuous drift.

## What Claude Code should NOT do

- **Don't edit `.weaver-sync/`.** It's machine-written.
- **Don't invent Convex `_id` values** in files. Reference other entities by slug.
- **Don't set runtime fields** in frontmatter (`created_at`, `updated_at`, `version`, `art_status`, `chat_thread_id`, `discovered_by` user_id).
- **Don't commit `.weaver-sync/` or generated art blob files.** If the user chooses to git-init the snapshot directory, these are gitignore material.
- **Don't assume the DB has changed between your runs.** Export fresh before important reconciliation.
- **Don't delete entity files to "delete" entities.** Deletion is a DB operation; propose it in plain English to the user. The importer's default is additive — a missing file does NOT cause DB deletion.

## Open questions

- **Branching.** Export of a multi-branch world — one directory per branch, or one directory with a branch axis? Probably per-branch-directory is cleaner; defer until branches ship (Wave 3).
- **Blob content in exports.** Exports reference blobs by hash, but don't include blob bytes by default. An `--include-blobs` flag could write blob files alongside for a fully-portable snapshot.
- **Large-scale imports.** 500+ locations in one import — batching strategy (per Convex mutation size limits) not yet specified. Wave 2 concern.
